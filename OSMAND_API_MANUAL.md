# OsmAnd AIDL API Integration Manual

This document serves as a comprehensive guide to integrating third-party Android applications with the OsmAnd map application via its AIDL (Android Interface Definition Language) API. It summarizes the required setup, configuration, and practical examples for interacting with OsmAnd, including importing and removing GPX data silently, and registering custom UI components like context menu buttons.

## Prerequisites

### 1. Enable the "OsmAnd development" Plugin
OsmAnd actively rejects third-party AIDL IPC commands (such as `importGpx` or `addContextMenuButton`) unless the **OsmAnd development** plugin is explicitly enabled within the OsmAnd app settings. Make sure this plugin is enabled in your target OsmAnd app (`net.osmand.plus` or `net.osmand`).

### 2. Avoid the AAR Dependency (ABI Mismatch)
To prevent `Binder invocation to incorrect interface` ABI mismatch errors, **do not** use the `android-aidl-lib` AAR dependency.
Instead, manually include the raw `.aidl` files and their corresponding Java stubs directly from the `r5.3` branch of the official [osmandapp/OsmAnd](https://github.com/osmandapp/OsmAnd) repository.

## Project Setup

### AndroidManifest.xml Configuration
To properly register your Android app as an OsmAnd plugin and allow it to query OsmAnd packages on newer Android versions (API 30+), your `AndroidManifest.xml` must include a `<queries>` block for the OsmAnd packages.

```xml
<manifest xmlns:android="http://schemas.android.com/apk/res/android"
    package="com.example.your_app">

    <queries>
        <package android:name="net.osmand.plus" />
        <package android:name="net.osmand" />
    </queries>

    <application>
        <!-- Your application components -->
    </application>
</manifest>
```

## Binding to the OsmAnd Service

When establishing the IPC connection to OsmAnd, you must bind to the **V2** service. Using the legacy V1 action will lead to connection rejections (`Binder invocation to incorrect interface`).

Always wrap your connection logic and intent parsing in `try/catch` blocks to prevent fatal application crashes if the remote service is unavailable.

```kotlin
fun bindService(application: Application): Boolean {
    try {
        val intent = Intent("net.osmand.aidl.OsmandAidlServiceV2")

        // Try OsmAnd+ first
        intent.setPackage("net.osmand.plus")
        var bound = application.bindService(intent, serviceConnection, Context.BIND_AUTO_CREATE)

        // Fallback to the free version if OsmAnd+ is not installed
        if (!bound) {
            intent.setPackage("net.osmand")
            bound = application.bindService(intent, serviceConnection, Context.BIND_AUTO_CREATE)
        }
        return bound
    } catch (e: Exception) {
        Log.e("OsmAndAidlHelper", "Error binding to OsmAnd Service", e)
        return false
    }
}
```

## API Operations

### GPX Management: Importing and Removing

To bypass GPX import dialog prompts and seamlessly overlay data, you can import and remove GPX tracks silently using the AIDL `importGpx` and `removeGpx` methods.

#### FileProvider and URI Permissions
To import GPX via OsmAnd AIDL reliably, pass the GPX data as a content `Uri` generated by a `FileProvider`.
**Crucial for Android 14+:** You must grant **both** `Intent.FLAG_GRANT_READ_URI_PERMISSION` and `Intent.FLAG_GRANT_WRITE_URI_PERMISSION` to the OsmAnd packages to prevent IPC import failures.

```kotlin
// Example FileProvider helper to generate the URI and grant permissions
fun getGpxUri(context: Context, fileName: String, gpxData: String): Uri {
    val file = File(context.cacheDir, fileName)
    file.writeText(gpxData)

    val uri = FileProvider.getUriForFile(context, "${context.packageName}.fileprovider", file)

    val flags = Intent.FLAG_GRANT_READ_URI_PERMISSION or Intent.FLAG_GRANT_WRITE_URI_PERMISSION
    context.grantUriPermission("net.osmand.plus", uri, flags)
    context.grantUriPermission("net.osmand", uri, flags)

    return uri
}
```

#### Importing a GPX
Once you have the `Uri` from the `FileProvider`, you can call `importGpx`:

```kotlin
val params = ImportGpxParams(
    getGpxUri(application, "my_track.gpx", "<gpx>...</gpx>"),
    "my_track.gpx",
    "red", // Optional color
    true   // Show on map
)
osmandService?.importGpx(params)
```

#### Removing a GPX
Removing a GPX layer is straightforward using the filename:

```kotlin
val params = RemoveGpxParams("my_track.gpx")
osmandService?.removeGpx(params)
```

### Registering Context Menu Buttons

You can inject custom buttons into the OsmAnd map context menu.

**Important:**
1. You must explicitly add a map layer via `addMapLayer` before you can bind context menu buttons to it.
2. The `layerId` parameter in both `AMapLayer` and `ContextMenuButtonsParams` must be a valid, non-empty string (e.g., `"my_custom_layer"`). If omitted or invalid, the IPC request will be rejected.

```kotlin
val layerId = "my_custom_layer"

// 1. Explicitly register the map layer
val layer = AMapLayer(layerId, "My Custom Layer", 5.5f, null)
osmandService?.addMapLayer(AddMapLayerParams(layer))

// 2. Define your buttons
val leftButton = AContextMenuButton(
    101, "Action 1", "Action 1", "ic_action_marker_dark", "ic_action_marker_dark", false, true
)
val rightButton = AContextMenuButton(
    102, "Action 2", "Action 2", "ic_action_marker_dark", "ic_action_marker_dark", false, true
)

// 3. Register the buttons using ContextMenuButtonsParams
val params = ContextMenuButtonsParams(
    leftButton,
    rightButton,
    "my_context_menu_id",
    application.packageName,
    layerId, // Must match the registered map layer ID!
    1L, // callbackId
    ArrayList<String>() // pointsIds
)

osmandService?.addContextMenuButtons(params, yourAidlCallbackStub)
```

## Icons

The icons defined by the OsmAnd developers are addressable by a name.
All the icons are in https://github.com/osmandapp/OsmAnd-resources/tree/master/rendering_styles/style-icons/poi-icons-png/drawable-xxhdpi 
The name to use is the file name in this location, without the prefix 'mx_' and without the file extension.

### Navigating to a Specific Map Location

When transitioning from your app back to OsmAnd to view a specific location, avoid using standard `geo:lat,lon` intents. OsmAnd intercepts `geo:` intents by triggering its internal "Search by Geolocation" reverse-geocoding overlay, which is notoriously slow and results in a long black screen or waiting period while the map transitions.

To instantly set the map location and bypass the search overlay, use the direct AIDL `setMapLocation` method:

```kotlin
fun setMapLocation(lat: Double, lon: Double, zoom: Int = 15): Boolean {
    try {
        // Create SetMapLocationParams: lat, lon, zoom, rotation, animated
        val params = net.osmand.aidlapi.map.SetMapLocationParams(lat, lon, zoom, 0f, true)
        return osmandService?.setMapLocation(params) ?: false
    } catch (e: Exception) {
        return false
    }
}
```

After setting the location via AIDL, simply launch OsmAnd's main intent (or call `finish()` if your app was launched transparently) to return the user to the newly focused map view immediately.

## Troubleshooting & Best Practices

* **Seamless Return to OsmAnd:** To return to the OsmAnd map programmatically after an action in your app, execute the launch intent: `packageManager.getLaunchIntentForPackage("net.osmand.plus")` or simply call `finish()`. Avoid doing heavy synchronous queries or using `postDelayed` handlers as they will introduce UI lag.
* **Testing Iterations:** When building for your application, ensure you maintain a consistent APK signature (e.g., using a static debug keystore) across builds. This prevents seamless update issues and ensures OsmAnd remembers your plugin configuration.
* **Location Parsers:** If your app intercepts data from OsmAnd (e.g., via `ACTION_SEND`), ensure your parsers can handle multiple text formats. OsmAnd may send data as URLs (`lat=X&lon=Y`) or as URIs (`geo:X,Y`).
